\subsection{Local Push-Task Generation and Evaluation}\label{subsec:push}

For the selected portal gap~$g$, PIHS generates a bounded set of local push
proposals and validates them by short-horizon simulation. A geometric proposal is
$\tau_{\mathrm{geo}}\triangleq(g,m,\mathbf{v})$, where~$m$ is the pushed movable
obstacle and~$\mathbf{v}$ is a short-horizon object velocity. Contact-mode
optimization augments it with a multi-robot mode~$\boldsymbol{\xi}$ to form a
simulated task. Thus, task generation is only a proposal mechanism; accepted
state transitions are always produced by physics rollouts.

Each side of~$g$ is treated as an independent full-opening option. For a side
object~$m$, the nominal displacement is chosen to make the gap width exceed~$W$
with a small safety margin. The cheaper side is sampled first, while the other
side remains a bounded backup when the preferred side is blocked or produces
insufficient progress.

\subsubsection{Recursive Push-Task Generation}
Task generation contains direct and recursive pushing. Direct pushing moves a
gap-side object. Recursive pushing creates auxiliary tasks for movable blockers
found along the swept motion. A lightweight reference polyline connects the
start, centers of previously opened gaps, centers of downstream gaps, and the
goal. This polyline is not a vehicle path and need not be collision free; it
only gives a corridor orientation for pushing obstacles away from both the
cleared passage and downstream gaps.

\textbf{Direct pushing.}
For a selected side object, the nominal linear direction is a randomized convex
combination of a gap-opening direction and a direction moving the object away
from the reference polyline. Around the nominal direction~$\bar{\mathbf{d}}$, a
small grid over linear-direction jitter and angular velocity is evaluated,
\begin{equation}\label{eq:jitter_sampling}
\mathcal{V}_{m}(\bar{\mathbf{d}})\triangleq
\{(v_m R(\theta)\bar{\mathbf{d}},\omega)
\mid \theta\in\Theta_g,\;\omega\in\Omega_g\},
\end{equation}
where~$v_m$ is a nominal object speed, $\Theta_g$ is a finite jitter-angle set,
and~$\Omega_g$ is a bounded angular-velocity grid. Each candidate velocity is
geometrically propagated until the gap width exceeds~$W$ or a local horizon is
reached. Directions are ranked by pushing feasibility and swept-region intrusion,
favoring motions that open the gap while preserving workspace for later rollouts.

\textbf{Recursive pushing.}
If the direct swept region intersects movable blockers, recursive tasks are
generated for those blockers. For each blocker, the nominal direction moves it
away from both the swept region and the reference polyline. The grid in
\eqref{eq:jitter_sampling} is reused, but the local goal is to move the blocker
out of the swept region rather than open the gap. Recursion creates a bounded
dependency chain in which middle or tail tasks clear blockers that prevent a
later root push.

\subsubsection{Contact-Mode Optimization}
For each proposal~$(g,m,\mathbf{v})$, the mode generator optimizes contacts in
the body frame of~$\Omega_m$. Candidate contacts are filtered by reachability and
positive normal motion along the desired local contact-point velocity. The
initial assignment greedily selects, for each robot, the feasible contact whose
unit generalized wrench best aligns with the residual friction demand; a local
replacement search then swaps contacts when the lower-bound feasibility score can
improve. The mode score is
\begin{equation}\label{eq:mode-feasibility}
\begin{aligned}
\mathrm{J}_{\texttt{F}}^m(\boldsymbol{\xi},\mathbf{v})
&\triangleq
\min_{\mathbf{q}_{m,\boldsymbol{\xi}}\in\mathrm{Q}_{m,\boldsymbol{\xi}}}
\left\|\mathbf{q}_{m,\boldsymbol{\xi}}+Q_\mu^m(\mathbf{v})\right\|_1,
\end{aligned}
\end{equation}
where~$\mathrm{Q}_{m,\boldsymbol{\xi}}$ is the admissible generalized-wrench set
under selected contacts and friction cones, and~$Q_\mu^m$ is the predicted
quasi-static friction wrench. The total loss combines~\eqref{eq:mode-feasibility}
with robustness and vertex-risk terms. A ModeTable stores previously validated
contact tuples indexed by object-shape features and desired velocity, and these
entries are reused as priors when enabled. Without a matching entry, the greedy
optimization above still produces and sorts feasible modes by loss. Each feasible
mode turns~$\tau_{\mathrm{geo}}$ into a simulated task
$\tau\triangleq(g,m,\mathbf{v},\boldsymbol{\xi})$.

\subsubsection{Push Simulation Evaluation}
Generated tasks are filtered by robot reachability, swept collision, and
expected clearance progress. The remaining tasks are evaluated by parallel
PyBullet rollouts,
\begin{equation}\label{eq:sim}
(\mathbf{s}',\rho)\triangleq\mathsf{EvalSim}(\mathbf{s},\tau),
\end{equation}
where~$\mathbf{s}'$ is the successor state and~$\rho$ records WCCG progress,
such as gap-width improvement or reduction of the estimated W--clearance
cost-to-go. Direct tasks are evaluated first. Recursive middle and tail tasks
are released only when no direct rollout is valid or direct progress is below a
threshold. PIHS keeps only physically validated successors for search expansion.
A task is accepted only if collision and contact-validity checks pass and the
rollout produces measurable gap or endpoint progress. Execution uses the same
primitive: a hybrid controller plans transition paths to pre-contact poses,
resolves conflicts by goal exchange or priority-based replanning, and switches to
contact tracking. During pushing, robots approach the current contact poses until
all are in contact, then track forward contact poses generated by an arc
transition of the pushed object.
